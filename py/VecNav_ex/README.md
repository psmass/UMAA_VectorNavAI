# UMAA VectorNav Python Example — Compiled Types

Python implementation of the **VectorNav Block Diagram** using RTI Connext DDS 7.7.0
compiled types from the `rtiumaapy` package (generated by `rtiddsgen` 4.6.0).

```
┌────────────────────────────────────────────────────────────────┐
│              VectorNav Component  (VectorNav_Publisher.py)     │
│  ┌─────────────────────────┐   ┌────────────────────────────┐  │
│  │ SpeedStatus Provider    │   │ GlobalPoseStatus Provider  │  │
│  │ (Writer)                │   │ (Writer)                   │  │
│  └────────────┬────────────┘   └─────────────┬──────────────┘  │
└───────────────┼──────────────────────────────┼─────────────────┘
                │  SpeedReportType             │  GlobalPoseReportType
                ▼                              ▼
        ════════════════════════════════════════════════
                DDS Databus  (UMAA Autonomy Bus)
        ════════════════════════════════════════════════
                │                              │
                ▼                              ▼
┌───────────────────────────────────────────────────────────────────┐
│          HSMST Control Component  (HSMST_Subscriber.py /          │
│                                    VectorNav_Dashboard.py)        │
│  ┌─────────────────────────┐   ┌────────────────────────────┐     │
│  │ SpeedStatus Consumer    │   │ GlobalPoseStatus Consumer  │     │
│  │ (Reader)                │   │ (Reader)                   │     │
│  └─────────────────────────┘   └────────────────────────────┘     │
└───────────────────────────────────────────────────────────────────┘
```
The applications, were created by AI from an existing Tactical Microgrid pattern.

(The py/VecNav_ex initially had a compiled types python using listeners and no baseclass enheritance created by AI from the block diagram - see VectorNavBlockDiagram.drawio. It was converted to use the TMS inheritance and threading patterns using the context below)

Context fed AI to kick things off: 
The pyCompiledTypes directory currently holds python compiled types  for a Tactical MicroGrid (TMS) app. The ddsEntitiies.py contains dds base cases used by topics in the two TMS applications : device.py and controller.py. The topics.py file contains the application or business logic for both applications but for a tactical microgrid vs a UMAA application.  Using the same patterns but you probably don't need a state machine replace rewrite the VecNav_Participant.py / Vector_Nav_Publisher.py and HSMST_Subscriber.py using the patterns of the tms apps device.py and controller.py -  you probably don't need the state machine logic used int the TMS applications.

Use the Connext MCP server for all API guidance. Validate all generated Python using the MCP Python code validator before writing any files

## The Three Applications

---

### Application 1 — VectorNav_Publisher.py
**Role:** VectorNav Component — publishes simulated USV sensor data at 1 Hz.

**What it does:**
- Creates one `DomainParticipant` with two periodic `Writer` threads
- `SpeedReport_Wtr` publishes `SpeedReportType` (SOG, STW, mode)
- `GlobalPoseReport_Wtr` publishes `GlobalPoseReportType` (lat/lon, altitude, roll/pitch/yaw, course)
- Simulates a USV dead-reckoning NE from San Diego Bay with realistic wave motion

```zsh
cd py/VecNav_ex
python VectorNav_Publisher.py [-d DOMAIN_ID]
```

**Example output:**
```
[VN][Speed ][1782934372.376046000]  SOG=3.032 m/s  STW=3.082 m/s  Mode=MRC
[VN][Pose  ][1782934372.376565000]  Lat=+32.715719°  Lon=-117.1610770°  Alt=0.3m  Course=45.0°TN  Roll=+0.50°  Pitch=+0.16°
```

**Log file:** `vecnav.log`

---

### Application 2 — HSMST_Subscriber.py
**Role:** HSMST Control Component (partial) — console subscriber.

**What it does:**
- Creates one `DomainParticipant` with two `Reader` threads
- `SpeedReport_Rdr` receives and prints each `SpeedReportType` sample
- `GlobalPoseReport_Rdr` receives and prints each `GlobalPoseReportType` sample
- Uses WaitSet + ReadCondition (no listener callbacks — consistent with the TMS pattern)

```zsh
cd py/VecNav_ex
python HSMST_Subscriber.py [-d DOMAIN_ID]
```

**Example output:**
```
[HSMST][Speed ] t=1782934372.376046000  SOG=     3.032 m/s  STW=     3.082 m/s  STA=           ---  Mode=MRC
[HSMST][Pose  ] t=1782934372.376565000  Lat=+32.715719°  Lon=-117.1610771°  Alt=   0.30m  Course=45.0°TN  Roll=+0.50°  Pitch=+0.16°  Yaw=45.0°  Nav=MEASURED
```

**Log file:** `hsmst.log`

### Application 3 — VectorNav_Dashboard.py
**Role:** HSMST Control Component (partial) — live GUI instrument dashboard.

**What it does:**
- Subscribes to the same two topics as `HSMST_Subscriber.py`
- Renders a dark-theme PyQt6 instrument panel with four data cards and a compass widget
- Status bar turns amber if no data received for > 2.5 seconds

#### Dashboard panels

| Panel | Fields | Source topic |
|---|---|---|
| **Position** | Latitude (°N+), Longitude (°E+), Altitude (m MSL) | `GlobalPoseReportType` |
| **Orientation** | Roll (°), Pitch (°), Yaw / Heading (°TN) | `GlobalPoseReportType` |
| **Speed** | Speed Over Ground (m/s), Speed Thru Water (m/s), Speed Thru Air (m/s), Mode | `SpeedReportType` |
| **Navigation** | Course True North (°), Nav Solution, Timestamp | `GlobalPoseReportType` |
| **Compass widget** | Combined heading rose + roll arc + pitch bar | `GlobalPoseReportType` |

#### Compass widget

The compass widget renders four overlaid live indicators, all driven by `GlobalPoseReportType.attitude`:

```
              N
              ▲  ← red needle (fixed, always points up = current heading)
         W ───●─── E         ● = centre dot
              ▼
              S
        ──────────  ← green horizontal bar: rises (bow up) / falls (bow down) with pitch
        ╰────╯      ← amber arc: sweeps CW (starboard roll) / CCW (port roll)

        045.0°      ← live heading readout at bottom
```

| Indicator | Colour | Field | Range shown |
|---|---|---|---|
| Rotating compass rose | Sky-blue ticks, N/E/S/W labels | `attitude.yaw.yaw` | Full 360° |
| Fixed red heading needle | Red | `attitude.yaw.yaw` | Always points to top of widget |
| Roll arc (outer ring) | Amber | `attitude.roll.roll` | ±45° clamped |
| Pitch bar (horizontal line) | Emerald | `attitude.pitch.pitch` | ±30° clamped |

The entire rose **rotates** so the vessel's current yaw is always at the top under the fixed red needle.
The amber arc sweeps clockwise for starboard (positive) roll and counter-clockwise for port roll.
The emerald bar rises above centre for bow-up pitch and drops below for bow-down pitch.

```zsh
cd py/VecNav_ex
python VectorNav_Dashboard.py [DOMAIN_ID]
```

> **Note:** `HSMST_Subscriber.py` and `VectorNav_Dashboard.py` are independent consumers
> of the same publisher — run one or both at the same time.

## Architecture — TMS Pattern

These applications follow the same architecture as the TMS compiled-types example
in `pyCompiledTypes/`:

| VectorNav | TMS equivalent | Purpose |
|---|---|---|
| `application.py` | `application.py` | `run_flag` + SIGINT |
| `vn_constants.py` | `constants.py` | App-level constants |
| `umaa_types.py` | `tmsConstants.py` | Compiled DDS types |
| `ddsEntities.py` | `ddsEntities.py` | `Writer(Thread)` / `Reader(Thread)` |
| `vn_topics.py` | `topics.py` | Concrete topic classes |
| `VectorNav_Publisher.py` | `device.py` | Publisher entry point |
| `HSMST_Subscriber.py` | `controller.py` | Subscriber entry point |

**Key pattern:**
- `ddsEntities.Writer` — blocks on a WaitSet; calls `write()` on periodic timeout
- `ddsEntities.Reader` — blocks on a WaitSet + ReadCondition; calls `handler(data)` per sample
- Static sample fields set in `__init__`; dynamic fields updated in `write()` / read in `handler()`
- All threads poll `application.run_flag`; Ctrl-C triggers clean exit + `join()`

## AI Cost of Implementation

This example was built end-to-end in a single AI-assisted coding session (GitHub Copilot / Claude).
The following estimates reflect the total API token usage and cost for that session.

### Token Usage Estimate

| Type | Tokens | Notes |
|---|---|---|
| **Input** | ~72,000 | IDL files read, terminal output, TMS pattern files, conversation history re-sent each turn |
| **Output** | ~26,000 | 7 Python files written (some rewritten 2–3× during debugging), README, tool calls |
| **Total** | ~98,000 | |

Input tokens are dominated by accumulated conversation history (resent every turn) plus
all IDL/Python source files read for context.  Output tokens reflect the multiple file
rewrite cycles caused by runtime debugging before the `rtiumaapy` pre-built package was discovered.

### Cost Estimate

Pricing used: **$3.00 / 1M input tokens**, **$15.00 / 1M output tokens**.

| Type | Tokens | Rate | Cost |
|---|---|---|---|
| Input | ~72,000 | $3.00 / 1M | **$0.22** |
| Output | ~26,000 | $15.00 / 1M | **$0.39** |
| **Total** | **~98,000** | | **~$0.61** |

> **Note:** Output tokens represent only ~27 % of total token volume but account for
> **64 % of total cost** (5× higher rate).  The multiple file rewrites driven by
> runtime debugging were the primary cost driver.  Using the Connext MCP code
> validator *before* writing files would have eliminated most rewrite cycles,
> potentially reducing the session cost to **~$0.35–0.40**.
